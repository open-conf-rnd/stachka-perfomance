# VPN Setup Checklist (Mac + FoXray + Xray)

## Рабочая настройка

- [ ] Сервер живой: `docker ps` показывает `amnezia-xray` в `Up`, порт `443/tcp` открыт.
- [ ] UUID клиента задан: в `server.json` есть актуальный `id` (UUID).
- [ ] REALITY-параметры совпадают: `serverName`, `publicKey (pbk)`, `shortId (sid)` одинаковы на сервере и в клиенте.
- [ ] FoXray профиль создан: `vless + reality`, `network=raw(tcp)`, `flow=xtls-rprx-vision`, `fingerprint=chrome`.
- [ ] Опции не ломают старт: `mux OFF`, `fragment OFF`, `tcpFastOpen OFF` (базовый режим).
- [ ] Профиль активирован: в FoXray профиль включен, `Real Ping` дает ответ.
- [ ] IP реально меняется: `curl -4 https://ifconfig.me` -> `38.100.139.39`.
- [ ] Гео подтверждается: `curl -s https://ipinfo.io/country` -> `NL`.
- [ ] Конфликтов нет: выключены/удалены лишние VPN-клиенты и фильтры (`v2RayTun`, старые профили, лишние прокси).
- [ ] Бэкап сохранен: рабочая `vless://...` ссылка и параметры (`UUID/pbk/sid/SNI`) сохранены.

## Сайт open-conf-rnd.pro через VPN

- **Через VPN документ быстрый (~0.7 s), а большой JS (416 KB) — ~60 s:** на пути до этого хоста через VPN очень низкая скорость скачивания (~7 KB/s). Не обрыв — запрос завершается, но браузер может резать по таймауту или показывать pending. Решение: по возможности без VPN; или уменьшить размер первого JS (code splitting / lazy-роуты), чтобы при такой скорости он успевал загрузиться.
- **curl в терминале грузится быстро, в браузере — pending / ERR_CONNECTION_RESET:** при нормальной скорости через VPN Copy as cURL отдаёт полный JS; маршрут default → utun4. Если же curl к большому JS с VPN даёт ~60 s — см. пункт выше (низкая throughput).
- Проверить: системный прокси (Настройки → Сеть → Прокси) — браузер должен использовать тот же выход, что и curl; отключить расширения; попробовать другой браузер.
- На сервере для статики включён **max_ranges 0** (nginx), чтобы при странных Range-запросах отдавать полный файл, а не 206 с 1 байтом.
- Если **curl к open-conf-rnd.pro с VPN висит** (и --http1.1, и --http2) — проблема не в браузере и не в HTTP/2, а в пути до хоста через VPN (соединение не устанавливается или обрывается). Проверить: `curl -4 -s --max-time 5 https://ifconfig.me` (VPN жив?); `curl -4 -s --max-time 15 https://open-conf-rnd.pro/` (таймаут?); тот же запрос без VPN (работает?).
- **Может ли быть Caddy или сертификаты?** Сертификаты — вряд ли (curl с VPN успешно делает TLS, в браузере нет ERR_CERT_*). Caddy — возможно: он отдаёт HTTP/2 (h2); сбой может быть в обработке h2 для Chrome через VPN. Проверка: в DevTools → Network включить колонку **Protocol** и посмотреть, h2 или http/1.1 у запросов к open-conf-rnd.pro. Тест из терминала с VPN: `curl -s -o /dev/null -w "%{time_total}s\n" --http1.1 https://open-conf-rnd.pro/assets/index-DNljJ0YL.js` и то же с `--http2` — если по отдельности оба ок, а в браузере только h2 и там зависает, гипотеза про Caddy/HTTP/2 усиливается.

## Быстрая диагностика (если снова проблемы)

- [ ] `curl` не меняет IP -> проверить, активен ли профиль и правильный ли `UUID`.
- [ ] Есть ping, но сайты долго грузятся -> проверить `fingerprint=chrome`, затем (при необходимости) включить только `tcpNoDelay`.
- [ ] Вообще не коннектится -> перепроверить `serverName/pbk/sid` (один символ ошибки ломает REALITY).
- [ ] После правок на сервере -> `docker restart amnezia-xray`.

## Как посмотреть, как ходит трафик

### 1. В браузере (Chrome DevTools)

- Открыть **Network** (Сеть), включить **Preserve log** и при необходимости **Disable cache**.
- Включить VPN, зайти на https://open-conf-rnd.pro/.
- Кликнуть по запросу **index-DNljJ0YL.js** (или документу) → вкладка **Timing**:
  - **Queueing** — сколько ждал в очереди (если много — браузер блокирует запрос другими).
  - **Stalled** — пауза до отправки (очередь, лимит соединений).
  - **DNS Lookup** — резолв домена.
  - **Initial connection** — установка TCP/TLS.
  - **SSL** — рукопожатие TLS.
  - **Waiting (TTFB)** — ожидание первого байта от сервера.
  - **Content Download** — приём тела ответа.

По тому, где большие числа, видно: очередь в браузере, медленный канал или сброс на этапе TLS/приёма.

- **Проверка «тот же запрос, что и браузер»:** по запросу к JS правый клик → **Copy** → **Copy as cURL**. Вставить в терминал и выполнить **при включённом VPN**. Если в терминале запрос завершается за сотни миллисекунд — проблема в том, как браузер держит/переиспользует соединение.

### 2. В терминале: имитация браузера (несколько запросов подряд)

С VPN включённым выполнить — покажет время на документ и на JS по одному соединению и «параллельно»:

```bash
# Один запрос — документ
time curl -s -o /dev/null -w "doc: %{time_total}s\n" https://open-conf-rnd.pro/

# Один запрос — JS (как браузер тянет ассет)
time curl -s -o /dev/null -w "js:  %{time_total}s\n" https://open-conf-rnd.pro/assets/index-DNljJ0YL.js

# Как браузер: документ + JS «параллельно» (два процесса)
(time curl -s -o /dev/null https://open-conf-rnd.pro/) &
(time curl -s -o /dev/null https://open-conf-rnd.pro/assets/index-DNljJ0YL.js) &
wait
```

Если по отдельности быстро, а при параллельном запуске один из запросов зависает — похоже на лимит/очередь на стороне VPN или маршрутизации.

### 3. Через какой интерфейс уходит трафик

```bash
# Маршрут до хоста сайта (с VPN должен быть через utun)
netstat -rn | head -5
ping -c 1 open-conf-rnd.pro   # посмотреть, через какой iface (на Mac в выводе не показывается, смотрим netstat)

# Пакеты на IP сайта (подставить свой utun — обычно utun3/utun4 при активном FoXray)
# sudo tcpdump -i utun3 -n host 91.188.213.251 -c 20
```

Сначала `netstat -rn`: шлюз по умолчанию при включённом VPN будет через **utun** (FoXray). Затем можно слушать этот интерфейс через `tcpdump`, чтобы убедиться, что запросы к 91.188.213.251 идут через него.
